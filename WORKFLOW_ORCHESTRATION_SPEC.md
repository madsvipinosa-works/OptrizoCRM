## Workflow Orchestration Spec (Lead → Proposal → Project → Milestones → Tasks)

This document defines the canonical workflow state transitions and the associated notification triggers for the Optrizo system.

### 1. Entities and State Machines

#### 1.1 Lead (CRM)
- Lead statuses in the system:
  - `New Inquiry`, `Contacted`, `Qualified`, `Proposal Sent`, `Negotiation`, `Won`, `Lost` (plus internal/legacy variants like `New`/`Completed` in some logic)
- Canonical lead transition events:
  - **Lead created**: when staff receives a website inquiry (`submitContactForm`), the lead starts in a “new inquiry” state.
  - **Proposal sent**: when staff sends a proposal email and/or updates proposal status to `Sent`, the lead transitions to `Proposal Sent`.
  - **Deal won**: when the lead is marked “Won” (or equivalently when the winning client accepts an approved proposal), the system provisions an operational project and a client portal user.
  - **Lost**: when staff marks the lead “Lost”, no project should be provisioned.

#### 1.2 Proposal (Commercial)
- Proposal statuses in the system:
  - `Draft`, `Sent`, `Approved`, `Rejected`
- Canonical proposal transitions:
  - **Draft → Sent**:
    - Triggered when staff sends the proposal to the lead and updates the proposal to `Sent`.
  - **Sent → Approved** (client-initiated):
    - Triggered when a logged-in `client` accepts the proposal.
    - Accepting a proposal triggers “deal won” provisioning logic for the lead.
  - **Sent → Rejected** (client-initiated):
    - Triggered when a logged-in `client` rejects the proposal.
    - System synchronizes lead status back to negotiation.

#### 1.3 Project (PM Engine)
- Project statuses in the system:
  - `Kickoff`, `In Progress`, `In Review`, `Completed`
- Canonical project lifecycle:
  - **Provisioned** during “deal won” workflow:
    - Created with status `Kickoff`.
  - **Updated** by admin/editor project actions:
    - Project status transitions should be coordinated with milestone/task transitions to avoid UI drift.

#### 1.4 Milestones (Project Roadmap)
- Milestone statuses in the system:
  - `Pending`, `In Progress`, `Client Approval`, `Completed`
- Canonical milestone workflow:
  - **Provisioned** during project provisioning:
    - Each default milestone starts as `Pending`.
  - **Admin/editor “Client Approval” gating**:
    - When a milestone is moved into `Client Approval`, all active tasks in that milestone are automatically blocked (`isBlockedByClient = true`, task status moved to `Blocked`).
  - **Client feedback drives milestone status** (client-initiated):
    - `REVISION_REQUESTED` ⇒ milestone status becomes `In Progress`
    - `APPROVED` ⇒ milestone status becomes `Completed`

#### 1.5 Tasks (Operational Work Units)
- Task statuses in the system:
  - `Todo`, `In Progress`, `Blocked`, `Done`
- Canonical task workflow:
  - **Blocked by client**:
    - Entered when milestone moves to `Client Approval` by admin/editor.
  - **Unblocked by client feedback**:
    - When client submits milestone feedback (approved or revision requested), tasks that are marked `isBlockedByClient = true` are unblocked and moved to `Todo`.
  - **Completion**:
    - Admin/editor updates tasks to `Done`.

### 2. Notification Triggers (Canonical)
This section lists the “source-of-truth” events that emit notifications.

#### 2.1 New inbound leads
- Trigger: lead created via contact form submission.
- Delivery:
  - Email notification to configured recipients.
  - In-app/admin notification for dashboard visibility.

#### 2.2 Proposal sent
- Trigger: staff sends a proposal email and marks the proposal `Sent`.
- Delivery:
  - Client receives an email with a login-based proposal viewing experience.
  - Admin notification can be generated by the proposal flow as needed.

#### 2.3 Proposal accepted / rejected
- Trigger: client acceptance or rejection.
- Delivery:
  - Admin notifications to surface lead/project changes.
  - Lead/proposal provisioning synchronization.

#### 2.4 Milestone feedback
- Trigger: client submits milestone feedback:
  - `REVISION_REQUESTED` or `APPROVED`
- Delivery:
  - Admin notifications indicating feedback outcome.
  - Milestone status updated deterministically.

#### 2.5 Overdue tasks
- Trigger: overdue task detection runs when the PM engine board page loads.
- Condition:
  - Task not done, due date exists and is in the past, and `overdueNotified = false`.
- Delivery:
  - System notifications to task assignees (if any).
  - Admin notifications for visibility.

### 3. Progress Computation Rule
- Client portal progress bar computation (current behavior):
  - `progressPercent = round( completedTasks / totalTasks * 100 )`
- Consistency requirement:
  - Admin dashboards that display progress must use the same definition to prevent “progress disagreement” across roles.

### 4. Authorization Coupling Notes
- Client-initiated actions that drive workflow:
  - proposal acceptance/rejection
  - milestone feedback submission
- These actions must enforce:
  - authenticated session
  - role `client`
  - ownership validation (proposal email match; milestone project stakeholder membership)
